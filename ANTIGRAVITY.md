# ANTIGRAVITY — Документация плагина Obsidian Google Drive Encrypted Sync

Этот файл содержит описание архитектуры плагина, ключевых технических решений, ограничений и исправлений, внесенных в процессе разработки, для будущих сессий pair programming.

---

## 📌 Общее описание
Плагин для Obsidian, обеспечивающий надежную двустороннюю синхронизацию заметок через Google Drive с **end-to-end шифрованием (E2EE)** на стороне клиента. Все имена файлов и их содержимое шифруются локально перед отправкой в облако.

---

## 🏗️ Архитектура и структура модулей

Код расположен в папке `src/` и разделен на следующие компоненты:

1. **`main.ts`** — точка входа плагина, инициализация модулей, управление командами, статус-баром и интеграцией с Obsidian.
2. **`sync-engine.ts`** ([sync-engine.ts](file:///c:/Users/antma/Documents/Personally/obsidian-google-plugin/src/sync-engine.ts)) — ядро синхронизации. Сопоставляет локальные и облачные файлы, строит план действий (скачивание, загрузка, удаление, разрешение конфликтов) и выполняет его.
3. **`crypto.ts`** ([crypto.ts](file:///c:/Users/antma/Documents/Personally/obsidian-google-plugin/src/crypto.ts)) — криптографический сервис:
   * **PBKDF2** для деривации ключа из пароля пользователя и соли.
   * **AES-256-GCM** для шифрования/дешифрования контента и путей к файлам.
   * **SHA-256** для хеширования (проверки целостности и изменений).
4. **`google-drive.ts`** ([google-drive.ts](file:///c:/Users/antma/Documents/Personally/obsidian-google-plugin/src/google-drive.ts)) — низкоуровневый клиент для работы с Google Drive API v3 через встроенный Obsidian метод `requestUrl` (минуя ограничения CORS).
5. **`state-tracker.ts`** — отслеживает состояние файлов (размеры, хеши, ID в Google Drive) и ведет базу данных синхронизации локально.
6. **`oauth.ts`** ([oauth.ts](file:///c:/Users/antma/Documents/Personally/obsidian-google-plugin/src/oauth.ts)) — менеджер авторизации через Google OAuth 2.0.
7. **`settings.ts`** ([settings.ts](file:///c:/Users/antma/Documents/Personally/obsidian-google-plugin/src/settings.ts)) — вкладка настроек плагина.

## 🛠️ Реализованные ключевые исправления и фичи

### 1. Безопасность при массовом удалении (Symmetric Deletion Guard)
* **Проблема**: Если пользователь случайно очищал локальный диск или ломалась база данных состояний, плагин мог расценить это как команду «удалить все файлы на Google Drive».
* **Решение**: Добавлен симметричный предохранитель в `sync-engine.ts`. Если план синхронизации предполагает удаление **более 5 файлов И более 50% всех синхронизированных заметок** в любом направлении (локально или на Google Drive), синхронизация останавливается с ошибкой. Для обхода требуется ручной запуск «Полной пересинхронизации».

### 2. Локальный HTTP-сервер для авторизации OAuth
* **Проблема**: Google OAuth 2.0 запрещает использовать кастомные протоколы (например, `obsidian://`) в качестве `redirect_uri` для десктопных приложений.
* **Решение**: Перевели OAuth на запуск временного HTTP-сервера на случайном порту loopback-интерфейса (`http://127.0.0.1:0`). При открытии окна авторизации плагин запускает сервер, перехватывает код и сразу выключает его.
* **Баг с `/favicon.ico`**: Браузер отправляет параллельно несколько запросов (например, за иконкой), у которых нет параметров авторизации, что приводило к ошибке `State mismatch error`. Было внесено исправление, которое игнорирует любые вспомогательные запросы и валидирует только реальный колбэк от Google.

### 3. Рекурсивное создание папок
* **Проблема**: При синхронизации глубоко вложенной структуры (например, `A/B/C/file.md`) на новое устройство, вызов `vault.createFolder("A/B/C")` падал с ошибкой, если папки `A` или `A/B` ещё не существовали.
* **Решение**: Метод `ensureDirectoryExists` был переписан так, что он последовательно проверяет и создает каждый уровень вложенности папок сверху вниз, гарантируя бесконфликтное восстановление структуры каталогов.

### 4. Ввод пароля через native Modal API
* **Проблема**: Создание окон ввода (`input`) прямо внутри DOM-дерева панели настроек приводило к тому, что Obsidian перехватывал клавиатурный фокус и горячие клавиши, делая невозможным ввод символов пароля.
* **Решение**: Разработан отдельный класс `PasswordModal` ([password-modal.ts](file:///c:/Users/antma/Documents/Personally/obsidian-google-plugin/src/ui/password-modal.ts)), наследующий стандартный `Modal` из API Obsidian. Внутри модальных окон Obsidian корректно передает фокус текстовым полям и отключает перехват клавиш.

### 5. Автоматическое сохранение и восстановление пароля
* **Проблема**: При перезапуске Obsidian или плагина криптографический ключ сбрасывался из памяти, и пользователю приходилось принудительно заходить в настройки и вводить пароль заново при каждой новой сессии, чтобы запустить синхронизацию.
* **Решение**: Пароль сохраняется локально в файле `data.json` хранилища (в свойстве `encryptionPassword`). При запуске Obsidian плагин автоматически восстанавливает и инициализирует крипто-сервис с помощью сохраненного пароля и соли, обеспечивая полностью автономную автосинхронизацию без участия пользователя.

### 6. Синхронизация соли шифрования (`.sync-salt`) для мультидевайса
* **Проблема**: Ключ шифрования генерируется на основе пароля И случайной соли. Если второе устройство генерировало свою собственную случайную соль, то сгенерированный ключ шифрования отличался даже при одинаковом пароле, из-за чего второе устройство не могло прочитать и расшифровать файлы с Google Drive (синхронизация возвращала `0/0`).
* **Решение**: Реализован механизм шаринга соли через Google Drive:
  * Первое устройство при первой настройке шифрования загружает свою соль на Drive в открытый файл `.sync-salt`.
  * Второе устройство при подключении или вводе пароля сначала пытается скачать `.sync-salt` из облака и инициализирует шифрование именно с этой солью. Это гарантирует генерацию абсолютно идентичных ключей на всех устройствах.

## 📋 Требования к Google Cloud Console
Для корректной работы авторизации и синхронизации:
1. **OAuth Client ID** должен быть создан с типом **Desktop application** (Десктопное приложение).
2. **OAuth Consent Screen** (экран согласия) должен находиться в режиме **Testing**, а e-mail пользователя должен быть добавлен в список **Test Users** (Тестовые пользователи).
3. Доступ запрашивается по области видимости **`https://www.googleapis.com/auth/drive.file`**, что исключает доступ плагина к файлам пользователя, которые не были созданы этим плагином.
